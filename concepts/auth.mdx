---
title: "Auth model"
description: "Long-lived API keys exchange for short-lived session tokens; session tokens authorize compute calls."
---

Mount Thor uses a two-tier credential model:

1. **API key** (`mthr_live_*`) — long-lived, customer-owned, returned exactly
   once at creation. You store it in a secret manager and never put it on a
   command line.
2. **Session token** (`mt_session_*`) — short-lived, derived from an API key,
   intended for compute calls. You request the TTL you need (300–3600
   seconds, default 900).

The split exists so a leaked process artifact (CI logs, kubeconfig file,
machine image) only ever exposes a session token with a finite blast radius,
not the long-lived API key.

## Lifecycle

<Steps>
  <Step title="Mount Thor mints the API key">
    First key is issued during account activation. Subsequent keys are minted
    with the self-service `POST /v1/admin/keys` route. The plaintext is shown
    once; Mount Thor stores only a one-way verifier plus the prefix and
    last-four characters for display.
  </Step>
  <Step title="You exchange the API key for a session token">
    `POST /v1/admin/sessions` with `Authorization: Bearer <api_key>`.
    Request a TTL in the 300–3600 second range; default is 900 (15 minutes).
  </Step>
  <Step title="You use the session token for compute">
    Pass it as a bearer token to `api.mountthor.com/v1/compute/*` or as the
    `--token` for `kubectl` against your tenant.
  </Step>
  <Step title="The session expires; you mint another">
    Sessions are not renewable. When one expires, present the API key again
    and mint a fresh session.
  </Step>
</Steps>

## API key creation

Self-service key management uses a long-lived Mount Thor API key with
`keys:read` or `keys:write` scope.

List key metadata:

```bash
curl https://api.mountthor.com/v1/admin/keys \
  -H "Authorization: Bearer $MT_API_KEY"
```

Create a key:

```bash
curl -X POST https://api.mountthor.com/v1/admin/keys \
  -H "Authorization: Bearer $MT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "display_name": "ci-prod",
    "scopes": ["sessions:write", "compute:read", "compute:write"],
    "expires_at": "2027-01-01T00:00:00Z"
  }'
```

Response:

```json
{
  "api_key_id": "<api key UUID>",
  "api_key": "mthr_live_01JZ8M7EXAMPLE",
  "plaintext_display": "once",
  "scopes": ["sessions:write", "compute:read", "compute:write"],
  "status": "active"
}
```

The plaintext `api_key` is returned exactly once. Mount Thor stores only a
one-way verifier plus the key prefix and last four characters.

Revoke a key:

```bash
curl -X DELETE https://api.mountthor.com/v1/admin/keys/{id} \
  -H "Authorization: Bearer $MT_API_KEY"
```

Key management exposes:

| Method | Route | Purpose |
|---|---|---|
| `GET` | `/v1/admin/keys` | List your keys (metadata only) |
| `POST` | `/v1/admin/keys` | Create a new key |
| `DELETE` | `/v1/admin/keys/{id}` | Revoke a key |

Reads return id, display name, scopes, status, creation time, and expiry. The
plaintext is never available after creation. Create-new-key plus delete-old-key
is the customer rotation flow.

## Session minting

```bash
curl -X POST https://api.mountthor.com/v1/admin/sessions \
  -H "Authorization: Bearer mthr_live_01JZ8M7EXAMPLE" \
  -H "Content-Type: application/json" \
  -d '{ "session_ttl_seconds": 900 }'
```

Response:

```json
{
  "token_type": "Bearer",
  "session_token": "mt_session_01JZ8M7EXAMPLE",
  "expires_at": "<RFC3339>",
  "scopes": ["compute:read", "compute:write"],
  "audience": "api.mountthor.com"
}
```

`session_ttl_seconds` is bounded to `[300, 3600]`. Sessions inherit the
parent API key's scopes — you cannot widen scope at exchange time.

## Rotation and revocation

- **API keys** rotate by creating a replacement key and deleting the old key
  with `DELETE /v1/admin/keys/{id}`.
- **API keys** are revoked with `DELETE /v1/admin/keys/{id}`.
- **Session tokens** expire on schedule. To stop future sessions from a key,
  revoke that API key.

## Scopes

Issued today:

| Scope | Allows |
|---|---|
| `compute:read` | Read compute resources in your tenant |
| `compute:write` | Create, update, and delete compute resources in your tenant |
| `sessions:write` | Mint short-lived Mount Thor session tokens |
| `keys:read` | Read API-key metadata |
| `keys:write` | Create and revoke API keys |
| `registries:read` | Read external registry grant status |
| `registries:write` | Create, update, and disable external registry grants |

API-key scopes determine which admin and compute calls a credential can make.

## Things that don't authenticate

Two endpoints on `api.mountthor.com` are intentionally unauthenticated:

- `GET /healthz` — returns `{"status":"ok"}` for liveness probes.
- `GET /openapi.json` — the public OpenAPI spec for the customer surface.

Everything else under `/v1/*` requires a valid bearer credential or a
single-use registration code.
