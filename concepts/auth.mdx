---
title: "Auth model"
description: "Long-lived API keys exchange for short-lived session tokens; session tokens authorize compute calls."
---

Mount Thor uses a two-tier credential model:

1. **API key** (`mt_live_*`) — long-lived, customer-owned, returned exactly
   once at creation. You store it in a secret manager and never put it on a
   command line.
2. **Session token** (`mt_session_*`) — short-lived, derived from an API key,
   intended for compute calls. You request the TTL you need (300–3600
   seconds, default 900).

The split exists so a leaked process artifact (CI logs, kubeconfig file,
machine image) only ever exposes a session token with a finite blast radius,
not the long-lived API key.

## Lifecycle

<Steps>
  <Step title="Mount Thor mints the API key">
    First key is issued during invite redemption. Subsequent keys are minted
    via `POST /v1/admin/customers/{customer_id}/api-keys` `[live]` (currently
    an admin-scoped route) or self-service via `POST /v1/admin/keys` `[todo]`.
    The plaintext is shown once; Mount Thor stores only an HMAC verifier
    plus the prefix and last-four characters for display.
  </Step>
  <Step title="You exchange the API key for a session token">
    `POST /v1/admin/sessions` (canonical) or `POST /v1/compute/sessions`
    (alias) with `Authorization: Bearer <api_key>`. Request a TTL in the
    300–3600 second range; default is 900 (15 minutes).
  </Step>
  <Step title="You use the session token for compute">
    Pass it as a bearer token to `api.mountthor.com/v1/compute/*` `[todo]`
    or as the `--token` for `kubectl` against your tenant namespace.
  </Step>
  <Step title="The session expires; you mint another">
    Sessions are not renewable. When one expires, present the API key again
    and mint a fresh session.
  </Step>
</Steps>

## API key creation

Currently mounted at the admin-scoped route `[live]`:

```bash
curl -X POST https://api.mountthor.com/v1/admin/customers/{customer_id}/api-keys \
  -H "Authorization: Bearer <admin token>" \
  -H "Content-Type: application/json" \
  -d '{
    "customer_user_id": "<customer user UUID>",
    "tenant_id": "<tenant UUID>",
    "display_name": "ci-prod",
    "scopes": ["compute:read", "compute:write"],
    "expires_at": "2027-01-01T00:00:00Z"
  }'
```

Response:

```json
{
  "api_key_id": "<uuid>",
  "api_key": "mt_live_01JZ8M7EXAMPLE",
  "plaintext_display": "once",
  "scopes": ["compute:read", "compute:write"]
}
```

The customer-self-service shape `[todo]` exposes:

| Method | Route | Purpose |
|---|---|---|
| `GET` | `/v1/admin/keys` | List your keys (metadata only) |
| `POST` | `/v1/admin/keys` | Create a new key |
| `DELETE` | `/v1/admin/keys/{id}` | Revoke a key |

Reads return id, display name, scopes, status, creation time, expiry. The
plaintext is never available after creation.

## Session minting

```bash
curl -X POST https://api.mountthor.com/v1/admin/sessions \
  -H "Authorization: Bearer mt_live_01JZ8M7EXAMPLE" \
  -H "Content-Type: application/json" \
  -d '{ "session_ttl_seconds": 900 }'
```

Response:

```json
{
  "token_type": "Bearer",
  "session_token": "mt_session_01JZ8M7EXAMPLE",
  "expires_at": "<RFC3339>",
  "scopes": ["compute:read", "compute:write"],
  "audience": "api.mountthor.com"
}
```

`session_ttl_seconds` is bounded to `[300, 3600]`. Sessions inherit the
parent API key's scopes — you cannot widen scope at exchange time.

## Rotation and revocation

- **API keys** can be rotated in place: `POST /v1/admin/customer-api-keys/{id}/rotate`
  atomically revokes the old key, mints a new plaintext, and revokes any
  active sessions tied to the old key.
- **API keys** can be revoked: `POST /v1/admin/customer-api-keys/{id}/revoke`.
  Active sessions tied to the revoked key are revoked at the same time;
  the response includes `active_sessions_revoked` so you know the blast radius.
- **Session tokens** expire on schedule and are not individually revocable
  via a customer-facing route in alpha. Mount Thor operators can revoke a
  customer-user's active session via
  `POST /v1/admin/customer-users/{customer_user_id}/session/revoke` if needed.

## Scopes

Issued today:

| Scope | Allows |
|---|---|
| `compute:read` | Read CRDs in your tenant namespace |
| `compute:write` | Create/update/delete compute CRDs in your tenant namespace |
| `compute:session` | Mint additional sessions (used by automation that bootstraps short-lived workers) |
| `compute` | Equivalent to `compute:read` + `compute:write` |

Mount Thor's permission model defines three customer access levels (Admin,
Operator, Reader). When [WorkOS identity setup](#identity-setup) is in
place `[todo]`, your IdP groups are mapped to these levels; until then,
scopes are managed by Mount Thor operators per-key.

## Identity setup (WorkOS) {#identity-setup}

`[todo]` — Mount Thor uses WorkOS for customer SSO and IdP group mapping.
After account activation, your customer identity admin is given a WorkOS
setup link to:

- Connect your existing IdP (Okta, Entra ID, Google Workspace, or another
  WorkOS-supported provider).
- Verify your customer domain.
- Map your IdP groups to Mount Thor access levels (Admin / Operator / Reader).

Mount Thor stores the WorkOS-returned **stable provider IDs** for each
selected group, not just display names — display names can change or
collide; provider IDs are the durable authorization input.

The WorkOS setup flow itself is documented under
[Onboarding](#) once it lands in alpha. In the interim, key issuance is
an out-of-band step coordinated with Mount Thor.

## Things that don't authenticate

Two endpoints on `api.mountthor.com` are intentionally unauthenticated:

- `GET /healthz` — returns `{"status":"ok"}` for liveness probes.
- `GET /openapi.json` — the public OpenAPI spec for the customer surface.

Everything else under `/v1/*` requires a valid bearer credential or a
single-use proof (an unexpired invite code, a signed WorkOS webhook
signature, or an OAuth state token from the SSO callback).
