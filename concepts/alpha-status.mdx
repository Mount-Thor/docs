Mount Thor is in alpha. The customer contract — routes, resource shapes,
behaviors — is the eventual product, but parts of it are still under the
current internal route shapes or pending implementation. These docs use
`[live]` / `[todo]` markers everywhere so you know what to expect.

<Note>
  If a route, field, or behavior is documented here without a marker, treat
  it as currently shipping. If it carries `[todo]`, the design is fixed but
  the implementation is in progress and the current internal shape is
  what's serving alpha traffic.
</Note>

## Customer-facing API edges

| Endpoint | Status | Notes |
|---|---|---|
| `https://api.mountthor.com` (public, bearer-auth) | `[live]` | Public NLB + Envoy edge in front of the admin API |
| `/healthz` | `[live]` | Liveness probe |
| `/openapi.json` | `[live]` | Generated from typed `utoipa` route metadata in admin-api |
| `/v1/admin/*` HTTP JSON | `[live]` for current shapes | Canonical customer-facing names are still being aligned |
| `/v1/compute/*` Kubernetes-compatible | `[todo]` at the public edge | Compute resources work today against your dedicated control plane |

Brokered transport at `access.mountthor.com` is the runtime endpoint for
emitted access commands; you do not call it directly.

## What changes when a marker flips

When a `[todo]` becomes `[live]`, the current internal route or behavior
remains available for at least one full alpha onboarding cycle to give
existing customers time to migrate. The aligned route is announced via
your Mount Thor primary contact and the docs page is updated atomically.
No silent route-shape changes.
