Mount Thor exposes Apple hardware at datacenter scale as programmable
infrastructure for AI workloads. You can reserve dedicated capacity or schedule
workloads on demand, either on bare-metal Apple hardware or in macOS virtual
machines. The Mount Thor API and CLI support full lifecycle management and
resource configuration.

<Note>
  Mount Thor is in alpha. These docs describe the supported customer API and
  compute workflow for the current customer program.
</Note>

## What's at the edge

| Endpoint | Protocol | Purpose |
|---|---|---|
| `https://api.mountthor.com/v1/admin/*` | HTTP JSON | Account activation, account status, keys, entitlements, sessions, registry grants |
| `https://api.mountthor.com/v1/compute/*` | Kubernetes-style API | Bare-metal and VM resources in your tenant |
| `https://api.mountthor.com/openapi.json` | HTTP JSON | OpenAPI reference for the admin surface |

Brokered transport for emitted bare-metal access commands and VM published-port
commands routes through `access.mountthor.com`, but you do not call
`access.mountthor.com` directly to create leases, VMs, sessions, or ports.

## Design principles

Bare-metal access is requested with `MacLease` and `MacAccessSession`. VM
access is requested on `VirtualMachine.spec.ports`. Host and VM access is never
standing: it derives from API resources, expires on a schedule, and is revoked
when the parent resource is deleted.

The API surface follows from those choices: customer-facing routes are typed
and versioned. If a route is reachable on `api.mountthor.com`, it is in the
[API reference](/api-reference/introduction). If a compute resource is
available in your tenant, it is in the compute OpenAPI reference and the
[Compute](/compute/bare-metal) guides.
