Mount Thor exposes bare-metal Macs and macOS virtual machines through a single
customer API at `api.mountthor.com`. You provision compute with a Kubernetes-shaped
control surface, present a Mount Thor session token, and get short-lived host
access — SSH, Screen Sharing, or a published TCP port — derived from declarative
resources. There is no shared hypervisor and no opaque pool fronting your work.

<Note>
  Mount Thor is in **alpha**. Some routes and resources are live today; others
  are part of the customer contract but not yet implemented. See
  [Alpha status](/concepts/alpha-status) for a precise breakdown, and look for
  the `[live]` and `[todo]` markers throughout these docs.
</Note>

## What's at the edge

| Endpoint | Protocol | Purpose | Status |
|---|---|---|---|
| `https://api.mountthor.com/v1/admin/*` | HTTP JSON | Account, registration, keys, entitlements, sessions | `[live]` for current shapes; `[todo]` for the canonical customer-facing names |
| `https://api.mountthor.com/v1/compute/*` | Kubernetes-compatible | Bare-metal and VM resources | `[todo]` at the public edge — compute resources work today against your dedicated control plane |
| `https://api.mountthor.com/openapi.json` | HTTP JSON | Generated OpenAPI spec for the admin surface | `[live]` |

Brokered transport for emitted bare-metal access commands and VM published-port
commands routes through `access.mountthor.com`, but you do not call
`access.mountthor.com` directly to create leases, VMs, sessions, or ports.

## Design principles

Mount Thor is dedicated, vertically integrated Apple Silicon. Every customer
gets their own Kubernetes control plane — not a namespace in a shared cluster.
Host access is never standing: it derives from an active `MacAccessSession` or
a `VirtualMachine.spec.ports` entry, both of which are declarative, expire on a
schedule you set, and revoke automatically when the parent resource is deleted.

The API surface follows from those choices: customer-facing routes are typed,
versioned, and generated from a registry; nothing about your fleet is implicit
or undocumented. If a route is reachable on `api.mountthor.com`, it is in the
[OpenAPI spec](/api-reference/introduction). If a CRD is queryable from your
control plane, it is in [Compute](/compute/bare-metal).
