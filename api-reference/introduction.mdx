Mount Thor exposes two customer API surfaces:

| Reference | Covers |
|---|---|
| Admin API | Account activation, account status, API keys, entitlements, sessions, and registry grants under `/v1/admin/*` |
| Compute API | Bare-metal leases, access sessions, VMs, image operations, and catalog reads under `/v1/compute/*` |

## What's in scope here

| | |
|---|---|
| **Audience** | Customer-facing admin routes and compute resources. |
| **Auth** | Admin routes use `mthr_live_*` API keys. Compute routes use `mt_session_*` session tokens minted from `/v1/admin/sessions`. Registration proves access through the single-use registration code. |
| **Contract** | The API reference is the field-level contract. Guide pages explain the recommended workflow. |

## What's not in scope here

- Routes that are not listed in this reference are not part of the customer
  API contract.

## Authentication

Every customer-facing route documented under this tab uses the
`api_key_bearer` security scheme:

```
Authorization: Bearer mthr_live_*  # long-lived API key
Authorization: Bearer mt_session_* # short-lived session token
```

See [Auth model](/concepts/auth) for which credential each route accepts and
how to mint sessions from API keys.

## Conventions

- Request and response bodies are JSON.
- Timestamps are RFC 3339 in UTC.
- IDs are UUIDs unless explicitly typed otherwise.
- Plaintext credentials (`mthr_live_*`, `mt_session_*`) are returned exactly
  once at issuance. Mount Thor stores only one-way verifiers and a limited
  display fingerprint.
- Errors return a stable machine-readable code; see [Common failures](/troubleshooting/common-failures).
