---
title: "Common failures"
description: "Failure codes you might encounter and how to recover from each."
---

Mount Thor returns stable error codes from the admin API and from compute
resources. Include the request ID or resource name when contacting support.

## Auth and account

| Signal | Meaning | What to do |
|---|---|---|
| `InvalidRegistrationCode` | The registration code is unknown, expired, revoked, or already redeemed. | Request a new registration email from Mount Thor. |
| `SessionExpired` | The session token has passed `expires_at`. | Mint a new session via `POST /v1/admin/sessions`. |
| `InsufficientScope` | The key or session lacks a scope the route requires. | Mint a session that includes the missing scope, or use a key with broader scopes. |
| `EntitlementDenied` | The account is not entitled to the requested product or class. | Check `GET /v1/admin/entitlements` or contact Mount Thor. |

## Bare metal (`MacLease`, `MacAccessSession`)

| Signal | Meaning | What to do |
|---|---|---|
| `NoCapacityAvailable` | No eligible host is available for the requested lease class. | Try another class, wait for capacity to free, or contact Mount Thor for capacity adjustment. |
| `MacAccessSession.status.phase=Expired` | The access session reached `expiresAt`. | Create a new `MacAccessSession`. The lease itself stays allocated. |
| `LeaseNotFound` (on `MacAccessSession`) | `spec.leaseRef` does not match an existing `MacLease`. | Verify the name, or create the lease first. |
| `LeaseNotReady` | Referenced lease exists but is not yet `Ready`. | Wait for the lease to reach `Ready` and create a new access session. |
| `UnsupportedSessionType` | `spec.type` is not `ssh` or `screen-sharing-over-ssh`. | Set `spec.type` to one of the supported values. |
| `PrincipalNotAllowed` | `spec.principal` is not allowed by the lease access policy. | Use an allowed customer principal or update the access policy. |
| `InvalidTTL` | `spec.requestedTTL` is unparseable. | Use a duration like `30m`, `2h`, or `8h`. |
| `TTLTooLong` | Requested TTL exceeds the per-session cap. | Request `8h` or less. |
| `AccessBoundaryError` | Mount Thor's access path is unavailable transiently. | Wait and retry; if persistent, contact Mount Thor. |
| `SessionRevoked` | The session was revoked before expiry. | Create a new `MacAccessSession`. |

## Virtual machines (`VirtualMachine`, `VirtualMachineImage`, image operations)

| Signal | Meaning | What to do |
|---|---|---|
| `ImagePullDenied` | Mount Thor cannot read the external image (registry credential expired, repository moved, etc.). | Verify the registry credential and image digest; rotate via `PATCH /v1/admin/registries/{id}`. |
| `ImageValidationFailed` | The VM image did not pass validation (digest mismatch, unsupported format, etc.). | Re-export the image with a fresh digest, then create a new `VirtualMachineImageOperation`. |

## Reading failure detail

For compute resource failures, the failure code lives on `status.failureCode`
and the human-readable message on the latest condition:

```bash
kubectl get macaccesssession bm-01-ssh \
  -o jsonpath='{.status.failureCode}{" "}{.status.conditions[-1:].message}{"\n"}'
```

For HTTP responses, Mount Thor returns:

```json
{
  "error": "<machine_code>",
  "message": "<human-readable>",
  "request_id": "<uuid>"
}
```

Always include `request_id` when contacting Mount Thor support — it's the
fastest path to an answer.

## When to escalate

Contact Mount Thor with the request ID, the resource (or HTTP route), and a
brief description if you see:

- `AccessBoundaryError` persisting beyond a few minutes.
- A `MacLease` stuck in `Pending` for longer than your typical capacity
  time without `NoCapacityAvailable`.
- An HTTP `500`-class response from `api.mountthor.com` (these should be
  exceptional; we'd like to hear about them).
