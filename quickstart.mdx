Mount Thor is a cloud platform making datacenter-scale Apple hardware
available, performant, and easy to use for reinforcement learning, agentic
deployments, and enterprise inference. Choose [bare metal](/compute/bare-metal)
for dedicated hardware, or use [virtual machines](/compute/virtual-machines)
for faster provisioning. This page walks you through account setup,
authentication, and launching your first machine.

<Note>
  Mount Thor is in alpha. APIs, resource names, and workflows may evolve as
  the platform improves.
</Note>

## Create Account

Mount Thor works directly with alpha customers to set up accounts. We will need
the following information:

- Primary contact email
- Preferred corporate display name
- Corporate legal name
- Email domain to allowlist, or an individual email if you are not using a
  corporate domain

After we create your account, we will send a registration link to your primary
contact email.

## Activate Account

Your primary contact email will receive a Mount Thor signup invite. The email
comes from `invites@fleet.mountthor.com` and includes your registration code
and a setup link. If you do not see it, check spam or quarantine for that
sender.

<Accordion title="View signup email example">

```text
From: invites@fleet.mountthor.com
Subject: Mount Thor signup invite for <Company>

You're invited to finish setting up <Company> on Mount Thor.

Open this link to activate your account:
https://api.mountthor.com/v1/admin/registrations

On the activation page, enter:
- Registration code: <registration-code>
- Your name: your full name.
- Work email: the email address that received this invite, unless your Mount Thor
  contact told you to use another approved work email.

This code expires in 24 hours. It can be redeemed once by an allowed email
address for this customer.

For next steps, visit:
https://docs.mountthor.com

If you did not request access to Mount Thor, please disregard this email.
```

</Accordion>

After you submit the registration form, Mount Thor activates your account and
displays your first API key.

<Warning>
  Store your API key in your secret manager before closing the page. The key is
  shown only once.
</Warning>

You can manage API keys in [List Keys](/api-reference/api-keys/list-keys):
list existing keys, create new keys for rotation, and revoke keys that are no
longer needed.

When compute access is ready, Mount Thor will send a confirmation email to your
account contact.

## Install the CLI

The `mthr` CLI is the primary interface for managing Mount Thor compute. Install
it:

```bash
curl -fsSL https://get.mountthor.com/install.sh | sh
mthr --version
```

## Authenticate

Log in through your browser with SSO:

```bash
mthr login --customer <CUSTOMER>
```

This opens your identity provider in a browser window. After you authenticate,
the CLI stores a session locally.

Check your identity:

```bash
mthr whoami
```

## Get Kubeconfig

Mount Thor exposes compute through the Kubernetes API. The `mthr kubeconfig`
command writes a kubeconfig that lets both `mthr` and `kubectl` talk to your
Mount Thor control plane.

```bash
mthr kubeconfig
```

By default, the command prompts for a save path (default `~/.kube/mount-thor`).
Use `--out <PATH>` to skip the prompt.

Set the namespace for your tenant:

```bash
export MOUNTTHOR_NAMESPACE="tenant-example"
```

## Provision a Bare-Metal Mac

Allocate a dedicated physical Mac with a single command. Mount Thor provisions
the hardware, installs the image, and prepares it for access.

Check available hardware classes and images:

```bash
mthr fleet images
```

Allocate a machine:

```bash
mthr bm allocate build-1 --class m4-24g --image xcode-16
```

This creates a `BareMetalMachine` resource in your Mount Thor control plane.
Wait for it to become ready, then SSH in:

```bash
mthr bm ssh build-1
```

The CLI automatically waits for the machine to be ready (up to 30 seconds by
default) and brokers a short-lived SSH grant — no separate access session
needed.

When you're done, release the machine:

```bash
mthr bm free build-1
```

Mount Thor revokes access, wipes the host, and returns it to the pool. For the
full bare-metal lifecycle, see [Bare Metal](/compute/bare-metal).

## Create a Virtual Machine

For faster create/delete cycles, create a virtual machine. Mount Thor places
the VM on eligible capacity, prepares the image, boots macOS, and publishes
your requested ports.

```bash
mthr vm create dev --image macos-15-dev-base --cpu 4 --memory 8 --disk 100 \
  --port ssh=tcp:22
```

Wait for the VM to start, then SSH in:

```bash
mthr vm ssh dev
```

Open a tunnel to an application port:

```bash
mthr vm tunnel dev --port-name app --local-port 8443
```

Delete the VM when you're done:

```bash
mthr vm delete dev
```

See [Virtual Machines](/compute/virtual-machines) for the full VM lifecycle,
desktop access, port publishing, and bootstrap data.

## Advanced: Direct Kubernetes Access

Every `mthr` command maps to a Kubernetes Custom Resource under the
`compute.mountthor.com/v1alpha1` API group. You can also manage resources
directly with `kubectl` using the kubeconfig from `mthr kubeconfig`.

List your bare-metal machines:

```bash
kubectl get baremetalmachines
```

List your VMs:

```bash
kubectl get virtualmachines
```

Browse the image catalog:

```bash
kubectl get images
```

See the [Bare Metal](/compute/bare-metal) and
[Virtual Machines](/compute/virtual-machines) guides for full CRD examples.

## Programmatic Access

For CI/CD and automation, use API keys with session tokens instead of
browser-based SSO.

Generate a session token:

```bash
export MT_API_KEY="mthr_live_01JZ8M7EXAMPLE"

curl -X POST https://api.mountthor.com/v1/admin/sessions \
  -H "Authorization: Bearer $MT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "session_ttl_seconds": 900 }'
```

Session tokens expire automatically (5–60 minutes, default 15 minutes). When a
token expires, generate a new one from your API key.

For headless CLI authentication (CI pipelines, automation), use principal-based
login:

```bash
mthr login --principal <PRINCIPAL_ID>
```

<Accordion title="Automate session token rotation">

Set `MT_API_KEY` from your secret manager:

```bash
export MT_API_KEY="mthr_live_01JZ8M7EXAMPLE"
```

Create the helper script:

```bash
cat > mt-refresh-session.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

: "${MT_API_KEY:?Set MT_API_KEY from your secret manager before running this script.}"

command -v jq >/dev/null || { echo "jq is required." >&2; exit 1; }
command -v kubectl >/dev/null || { echo "kubectl is required." >&2; exit 1; }

ttl_seconds="${MT_SESSION_TTL_SECONDS:-900}"
kubectl_user="${MT_KUBECTL_USER:-mount-thor-session}"
KCFG="${KCFG:-${HOME}/.kube/mount-thor}"
mkdir -p "$(dirname "$KCFG")"

response="$(
  curl -fsS -X POST https://api.mountthor.com/v1/admin/sessions \
    -H "Authorization: Bearer ${MT_API_KEY}" \
    -H "Content-Type: application/json" \
    -d "{\"session_ttl_seconds\": ${ttl_seconds}}"
)"

session_token="$(printf '%s' "$response" | jq -r '.session_token')"
expires_at="$(printf '%s' "$response" | jq -r '.expires_at')"

if [[ -z "$session_token" || "$session_token" == "null" ]]; then
  echo "Mount Thor did not return a session token." >&2
  exit 1
fi

kubectl --kubeconfig "$KCFG" config set-credentials "$kubectl_user" \
  --token="$session_token" >/dev/null

printf 'export MT_SESSION_TOKEN=%q\n' "$session_token"
printf 'Session token expires at %s\n' "$expires_at" >&2
EOF
```

Make it executable and run it:

```bash
chmod +x mt-refresh-session.sh
source <(./mt-refresh-session.sh)
```

</Accordion>
